% ── Abstract ────────────────────────────────────────────────────────────────
\noindent
Il progetto realizza una \textbf{piattaforma Big Data end-to-end} per l'analisi del traffico di rete.
L'infrastruttura simula un ambiente enterprise attraverso \textbf{Kathara} (container Docker) e integra
\textbf{Apache Spark} per l'elaborazione distribuita di oltre 66 milioni di record, \textbf{MongoDB}
come Data Lake operazionale e modelli \textbf{LLM locali} (Ollama) per l'assistente AI.
Il tutto è fruibile tramite una dashboard \textbf{Streamlit} accessibile via browser.

\section{Architettura dell'Infrastruttura}

\subsection{Topologia di Rete}
La rete è simulata su Kathara e organizzata in \textbf{tre Sistemi Autonomi (AS)}.
\textbf{AS1} ospita la rete locale con i client e il nodo attacker per i test di sicurezza.
\textbf{AS2} è il Data Center, cuore del sistema: implementa un fabric \textbf{Clos EVPN-VXLAN}
(Top-of-Fabric, Spine, Leaf) e ospita tutti i servizi applicativi.
\textbf{AS3} simula la connettività Internet esterna con DNS autoritativo.
Il router \textbf{r5} funge da gateway perimetrale del Data Center e ospita il firewall dinamico.

\section{Il Dataset: BigFlow-NIDS}

\subsection{Origine e Caratteristiche}
Il dataset \textbf{BigFlow-NIDS} è disponibile su Kaggle e raccoglie flussi di traffico etichettati
per il rilevamento delle intrusioni di rete (NIDS). Contiene oltre \textbf{66.355.798 record}
suddivisi tra traffico legittimo (\texttt{Benign}) e diverse categorie di attacco (DDoS, Brute Force, ecc.).

Il dataset viene convertito da CSV a \textbf{Apache Parquet} tramite Spark al primo avvio
(Listato~\ref{lst:ingest}). Parquet garantisce compressione columnar, schema-on-read e
compatibilità nativa con Spark SQL senza parsing aggiuntivo.

\begin{lstlisting}[style=pystyle, caption={Ingestione CSV e conversione in Parquet}, label=lst:ingest]
df = spark.read \
     .option("header", "true") \
     .option("inferSchema", "true") \
     .option("samplingRatio", "0.1") \
     .csv("/opt/spark/data/raw/BigFlow-NIDS.csv")
df = df.withColumn("label", df["label"].cast("string"))
df.write.mode("overwrite") \
     .parquet("/opt/spark/data/processed/BigFlow-NIDS.parquet")
\end{lstlisting}

\section{MongoDB come Data Lake}

\subsection{Motivazione della Scelta}
MongoDB svolge il ruolo di \textbf{Data Lake operazionale} per i dati live e semi-strutturati,
complementare al Parquet storico. I motivi principali della scelta sono:
\begin{itemize}[noitemsep]
  \item \textbf{Schema flessibile}: documenti a struttura variabile, ideali per stream PCAP e alert;
  \item \textbf{Capped Collections}: buffer circolari nativi a dimensione fissa per dati live;
  \item \textbf{Connettore Spark}: \texttt{mongo-spark-connector\_2.12-10.3.0.jar} per Data Federation;
  \item \textbf{Ricerca full-text}: filtri \texttt{\$regex} per il Data Catalog.
\end{itemize}

\subsection{Capped Collection per il Traffico Live}
La collezione \texttt{live\_traffic} è di tipo \textbf{Capped}: struttura circolare in cui i documenti
più vecchi vengono sovrascritti automaticamente al raggiungimento del limite configurato,
mantenendo costante la dimensione del database:

\begin{lstlisting}[style=pystyle, caption={Creazione della Capped Collection}]
db.create_collection(
    "live_traffic",
    capped=True,
    size=10 * 1024 * 1024,  # 10 MB
    max=5000                 # max 5000 documenti
)
\end{lstlisting}

Ogni documento inserito ha struttura: \texttt{timestamp}, \texttt{summary} (descrizione Scapy),
\texttt{length}, \texttt{src}, \texttt{dst}, \texttt{proto}.

\section{Pipeline di Ingestione Dati}

\subsection{Ingestor PCAP in Tempo Reale}
Il modulo \texttt{pcap\_ingestor.py} monitora in polling (ogni secondo) il file
\texttt{analisi\_traffico.pcap} prodotto dal nodo sniffer. Il suo comportamento è descritto
nell'Algoritmo~\ref{alg:ingestor}.

\begin{algorithm}[H]
\caption{PCAP Ingestor}
\label{alg:ingestor}
\begin{algorithmic}[1]
\State $ultimo \gets 0$
\While{True}
  \State $pacchetti \gets \Call{leggiPCAP}{\textit{pcap\_path}}$
  \If{$|pacchetti| < ultimo$}
    \State $ultimo \gets 0$ \Comment{Rilevata rotazione del file}
  \EndIf
  \If{$|pacchetti| > ultimo$}
    \State $nuovi \gets pacchetti[ultimo:]$
    \For{ogni pacchetto $p$ in $nuovi$}
      \State Estrai \textit{src, dst, proto, summary}
      \If{$src \in black\_list$ \textbf{and} $src \notin blocked$}
        \State Invia \texttt{BLOCK:src} via UDP al firewall su r5
        \State Inserisci in \texttt{blocked\_ips}
      \EndIf
      \State Analizza pattern TCP (BGP, FTP, SSH, HTTP, MongoDB)
      \If{pattern rilevato \textbf{and} $src \notin white\_list$}
        \State Inserisci alert in \texttt{alerts} (CRITICAL/INFO)
      \EndIf
    \EndFor
    \State Inserisci $nuovi$ in \texttt{live\_traffic}
    \State $ultimo \gets |pacchetti|$
  \EndIf
  \State \Call{attendi}{1 s}
\EndWhile
\end{algorithmic}
\end{algorithm}

\subsection{Data Federation Spark + MongoDB}
Il \textbf{Data Lake Explorer} monta due View federate interrogabili con una singola query SQL:
\texttt{storico\_parquet} (file Parquet) e \texttt{live\_mongo} (collezione MongoDB).
Esempio di join cross-source:

\begin{lstlisting}[language=SQL, caption={Esempio di query federata Parquet + MongoDB}]
SELECT s.IPV4_SRC_ADDR, l.summary
FROM storico_parquet s
JOIN live_mongo l ON s.IPV4_SRC_ADDR = l.src
LIMIT 10
\end{lstlisting}

\section{Analisi del Traffico: Query Spark SQL}

Tutte le analisi vengono eseguite via Spark SQL su oltre 66 milioni di record.
I file \texttt{.sql} risiedono in \texttt{storage/src/analytics/queries/} e
vengono caricati dinamicamente dalla dashboard, separando la logica applicativa
dalle interrogazioni dati.

\subsection{Classificazione degli Attacchi per Vettore}

La query \texttt{rilevamento\_anomalie.sql} classifica i flussi malevoli usando
una logica rule-based su porte di destinazione e volumi di traffico, espressa
come CTE (Listato~\ref{lst:anomalie}).

\begin{lstlisting}[language=SQL, caption={Classificazione rule-based dei flussi anomali}, label=lst:anomalie]
WITH traffico AS (
    SELECT
        CASE
            WHEN l4_dst_port = 179 OR l4_src_port = 179
                THEN 'BGP Hijacking/Exploit'
            WHEN l4_dst_port = 22  THEN 'SSH Brute Force'
            WHEN l4_dst_port = 53  THEN 'DNS Amplification'
            WHEN l4_dst_port IN (80, 443)
                THEN 'Web Attack (HTTP/S)'
            WHEN l4_dst_port = 21  THEN 'FTP Exploit'
            WHEN flow_duration_milliseconds < 100
                 AND in_pkts > 1000 THEN 'L3/L4 Flood (DDoS)'
            WHEN in_pkts > 5000 AND in_bytes > 100000
                THEN 'Volumetric DDoS'
            ELSE 'Altro / Malicious General'
        END AS vettore_attacco,
        label AS etichetta_dataset,
        COUNT(*) AS occorrenze,
        ROUND(AVG(flow_duration_milliseconds), 2)
            AS durata_media_flusso,
        ROUND(SUM(in_pkts + out_pkts), 0) AS pacchetti_totali,
        SUM(in_bytes + out_bytes) AS traffico_b
    FROM traffico_nids
    WHERE label != 'Benign'
    GROUP BY 1, 2
)
SELECT vettore_attacco, etichetta_dataset, occorrenze,
       durata_media_flusso, pacchetti_totali, traffico_b,
       CASE
           WHEN traffico_b >= 1099511627776
               THEN CONCAT(ROUND(traffico_b/1099511627776.0,2),'TB')
           WHEN traffico_b >= 1073741824
               THEN CONCAT(ROUND(traffico_b/1073741824.0,2),'GB')
           WHEN traffico_b >= 1048576
               THEN CONCAT(ROUND(traffico_b/1048576.0,2),'MB')
           WHEN traffico_b >= 1024
               THEN CONCAT(ROUND(traffico_b/1024.0,2),'KB')
           ELSE CONCAT(traffico_b,'B')
       END AS traffico_h
FROM traffico
ORDER BY traffico_b DESC
\end{lstlisting}

La colonna \texttt{traffico\_h} converte automaticamente i byte in formato
leggibile (KB, MB, GB, TB) tramite soglie potenze di 1024.
Il grafico risultante (Figura~\ref{fig:anomalie}) mostra i vettori d'attacco
ordinati per volume di traffico totale generato.

\subsection{Top 5 Attaccanti}

La query \texttt{top\_attaccanti.sql} identifica i cinque IP sorgente responsabili
del maggior numero di flussi malevoli:

\begin{lstlisting}[language=SQL, caption={Top 5 IP attaccanti per numero di flussi}]
SELECT ipv4_src_addr AS ip_attaccante,
       COUNT(*)              AS occorrenze,
       SUM(in_bytes + out_bytes) AS traffico_b,
       label AS classe
FROM traffico_nids
WHERE label != 'Benign'
GROUP BY ipv4_src_addr, label
ORDER BY occorrenze DESC
LIMIT 5
\end{lstlisting}

Dalla dashboard, un click sull'IP nella tabella risultante trasmette via UDP
il comando \texttt{BLOCK:<ip>} al demone firewall su r5.

\subsection{Streaming e Separazione del Reasoning}

t_match = re.search(
    r'<think>(.*?)(?:</think>|$)', raw_stream, re.DOTALL)
if t_match:
    thinking_text = t_match.group(1)

    r'<think>.*?(?:</think>|$)', '', raw_stream, flags=re.DOTALL)
\end{lstlisting}

\subsection{RAG Contestuale}
Per le domande riguardanti la sicurezza, l'AI Copilot implementa una forma
semplificata di \textbf{Retrieval-Augmented Generation}: prima di invocare il
modello, il sistema recupera da MongoDB gli ultimi 5 IP bloccati e i 5 alert più
recenti, iniettandoli nel prompt come contesto reale del Data Center.
L'algoritmo di selezione è descritto nell'Algoritmo~\ref{alg:rag}.

\begin{algorithm}[H]
\caption{RAG contestuale per l'AI Copilot}
\label{alg:rag}
\begin{algorithmic}[1]
\State $kw \gets [\text{"ip", "bloccat", "firewall", "traffico", ...}]$
\State $greetings \gets [\text{"ciao", "hello", "chi sei", ...}]$
\State $generic \gets (len(prompt) \leq 15$ \textbf{or} $prompt \in greetings)$
\If{$generic$}
  \State $prompt\_finale \gets \text{"Rispondi in italiano: "} + prompt$
\Else
  \State $blocked \gets \Call{MongoDB.blocked\_ips.find}{\{"status":"BLOCKED"\}, limit=5}$
  \State $alerts \gets \Call{MongoDB.alerts.find}{\{\}, sort=\{ts:-1\}, limit=5}$
  \State $contesto \gets \Call{formatta}{blocked, alerts}$
  \State $prompt\_finale \gets contesto + \text{"\textbackslash n Domanda: "} + prompt$
\EndIf
\State \Return $\Call{OllamaStream}{model, prompt\_finale, system}$
\end{algorithmic}
\end{algorithm}

\section{Sistema di Sicurezza e Risposta Automatica}

Il sistema implementa un \textbf{ciclo chiuso di rilevamento e risposta} illustrato
in Figura~\ref{fig:ciclo}. Il modulo \texttt{pcap\_ingestor.py} genera alert in
MongoDB; la dashboard espone il pulsante di blocco; il comando raggiunge via UDP
il demone \texttt{firewall\_daemon.py} su r5 che applica regole \texttt{iptables}.
Lo script \texttt{sync\_firewall.py}, eseguito all'avvio, ripristina tutte le regole
da MongoDB garantendo la sopravvivenza al riavvio (\textbf{state recovery}).

\section{Conclusioni}

Il progetto ha dimostrato l'applicazione pratica delle principali tecnologie Big Data
in uno scenario realistico:

\begin{itemize}[noitemsep]
  \item \textbf{Ingestion eterogenea}: dataset storico Parquet (66M+ record) e dati live PCAP,
        entrambi interrogabili via Spark SQL grazie alla Data Federation.
  \item \textbf{Elaborazione distribuita}: Apache Spark con query SQL complesse su aggregazioni,
        window functions temporali e join tra sorgenti dati eterogenee.
  \item \textbf{Visualizzazione analitica}: sei visualizzazioni interattive Altair
        (barre, line chart multi-serie, heatmap) offrono prospettive complementari
        sul traffico di rete.
  \item \textbf{Architettura Data Lake}: MongoDB come Data Lake operazionale
        (Capped Collection per stream live, Data Catalog per i metadati).
  \item \textbf{Integrazione AI}: LLM locali con RAG contestuale su dati MongoDB,
        prompt engineering e generazione automatica di query SQL (NL2SQL),
        senza dipendenza da servizi cloud.
\end{itemize}

\paragraph{Sviluppi futuri.}
Integrazione di un modello Spark MLlib (Random Forest o XGBoost) per la
classificazione automatica del traffico; sistema di alerting proattivo basato
su soglie dinamiche calcolate con Spark Structured Streaming;
configurazione multi-worker Spark su nodi distinti per testare lo scaling orizzontale.
